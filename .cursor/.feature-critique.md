# Feature Critique: qavis-advisory gate in devkit ship / pre-commit

**Verdict**: RETHINK
**Date**: 2026-07-07
**Proposal**: Add a `qavis-advisory` gate so `devkit ship` (strict) soft-blocks and pre-commit nudges when a diff "deserves qavis QA", chosen by a bench-selected classifier.

## Executive Summary
The need is real (UI PRs ship un-QA'd because qavis only fires on explicit request) and the bench-first / exit-code discipline is right for this codebase. But the flagship behaviour — the strict-ship exit-3 soft-block — is **unsatisfiable by compliance in v1**: at commit time no PR exists, so the marker check always misses, every UI ship exit-3 blocks, and the only escape is an override (running qavis cannot clear it because the pass-receipt is deferred). That plus a bench `--fail` contract that contradicts a recorded Target, and a layer-coupling smell against the fallow precedent, means the mechanism needs redirecting before code — not just patching.

## Feasibility Assessment
- **Status**: Partially Feasible
- **Evidence**: Pre-commit CAN read `git diff --cached` + `.qavis/recipe.json` (completeness.mts:177, qavis recipe.ts:35). The regex mirror is accurate (qavis route.ts:11-19; the plan's trap categories hold — `.ts`/`packages/design/` miss UI_EXT/UI_PATH → false-neg; `components/Button.test.tsx` hits both → false-pos). Sentry's import-from-gate no-drift pattern is real (check-sentry.mts:130-250, bench.mts:36).
- **Blockers**: The v1 clear-path (below) — the mechanism as written cannot be satisfied by doing the thing it asks for.

## Critical Issues (Blockers)
1. **The strict-ship exit-3 block is unsatisfiable by compliance in v1 (always-block on every UI ship).**
   - Problem: `deservesQa = isUiAffecting && hasRecipe && !hasQavisMarker(existingBody)`. On a *new* ship the PR doesn't exist, so `existingBody` is empty and the marker is always absent → ADVISE for every UI change in a recipe repo → exit 3 under `GUARD_AI_STRICT` (set by commit-with-gate-capture.sh:51). The plan explicitly *defers* the pass-receipt ("upgrade path"), so **running qavis cannot clear the block** — only `GUARD_QAVIS_OK` / bypass can.
   - Evidence: plan §1/§4 + commit-with-gate-capture.sh:43-51; qavis has no local pass-receipt today.
   - Impact: The flagship path (`devkit ship` of a UI branch) becomes a mandatory override on every ship. A gate that can only be bypassed, never satisfied, trains reflexive bypass — the dead-gate anti-pattern check-sentry.mts:14-16 names explicitly. This invalidates the core mechanism *as written*.
   - Alternative: Pull the staged-diff-SHA pass-receipt INTO v1 (mirror the review-gate PASS cache: review-gate-in-chain records `.devkit/review-cache.json` keyed by staged-diff SHA). Then `qavis qa` on the staged tree writes a receipt the gate reads → a real soft-block-until-run. If that's too big for v1, make ship **WARN-by-default** (exit 0 + stderr) like completeness's actual FAIL contract (completeness.mts:242-249) and revisit exit-3 when the receipt lands.

2. **The plan misreads the "completeness contract, reused verbatim".**
   - Problem: Completeness's exit 3 is the *judge-outage-under-strict* case (fail-closed on could-not-run), NOT a positive finding — a completeness FAIL verdict is WARN-only (exit 0) even under strict unless `COMPLETENESS_HARD` (completeness.mts:229-249). The plan maps a *positive ADVISE* to exit 3, inverting the contract and violating the "LLM never CREATES a block" invariant for the llm/hybrid modes.
   - Evidence: completeness.mts:15-20, 242-249; check-sentry.mts:14-16, 204-206.
   - Impact: A nondeterministic judge becomes the sole creator of a ship hard-stop. (Regex mode is deterministic, so a deterministic block there is defensible like a ratchet — but the plan's llm/hybrid modes are not.)
   - Alternative: If you keep exit-3, gate it on the *deterministic* regex only (never the LLM verdict alone), and give the husky fragment its own exit-3 copy (see W1).

## Warnings (Non-blocking but significant)
1. **exit-3 husky fragment messaging collision.** The plan says add `GUARD_FRAGMENTS.qavisAdvisory` with "same rc handling as decisions/review". Those fragments render exit 3 as "judge unavailable — strict ship mode failed closed. Check claude auth/quota" (husky-block.mts:49-53, 67-69). A positive-ADVISE exit 3 would print that misleading remedy. The fragment needs its own copy → not "same as decisions/review".
2. **Trailer override `qavis: skip` is dead at pre-commit.** A pre-commit hook never receives the commit message (that's why sentry & completeness live at commit-msg — check-sentry.mts:18, completeness.mts:2-3). Only the env overrides (`GUARD_QAVIS_OK`, `GUARD_NO_QAVIS_ADVISORY`) will work. Drop the trailer, or move the gate to commit-msg (which then loses the clean pre-commit staged-file model and needs commit-msg wiring devkit does not generate — see W3).
3. **Wiring is under-specified and won't propagate as written.** husky-block builds pre-commit blocks from `AI_GUARD_IDS` (line 82), `STANDALONE_GATES` (184), and consumers select via `components.mts` `GUARD_IDS`/`RECOMMENDED_GUARD_IDS` (12-19). Adding only `GUARD_FRAGMENTS.qavisAdvisory` propagates nowhere. Also note: **sentry is a bin that is NOT wired into any consumer hook** (grep confirms), and devkit generates **no commit-msg hook** — so completeness/sentry auto-wiring doesn't exist; the plan's "add a fragment" claim overstates the precedent. Enumerate: AI_GUARD_IDS + STANDALONE_GATES + GUARD_IDS + the descriptor list + doctor drift, and honour `bunx guard-*` (consumer) vs `node gate-engine/…` (devkit self-dogfood, devkit-self-dogfood Target).
4. **Bench `diff` vs gate `git diff --cached` drift.** For no-drift the gate must force `-c diff.noprefix=false -c diff.mnemonicPrefix=false` (completeness.mts:207, detect Target) so a consumer's git config can't change the header bytes the shared classifier sees vs the authored corpus.
5. **`qavis qa … --annotate description` (no --pr) is a broken remedy.** qavis annotates only on a PASS and only with `--pr` (cli.ts:47-49, 140-151). At commit time there's no PR → the printed fix is a silent no-op. Fix the advisory copy.
6. **Corpus: label circularity + nothing to mine from devkit.** Mining "qavis was historically run" ≠ "should have been QA'd" (circular). And devkit is `frontendRoots: []` — a CLI with essentially zero UI diffs — so the ADVISE class can only come from qavis history or authored traps. ~30-40 rows across ~8 categories (~4-5/row) is enough to *demonstrate* the regex-fails-the-traps bake-off but too thin for a durable McNemar `--fail` floor. Treat the bench as a one-time bake-off tool, or grow the mined set.
7. **Scope: three classifier modes before the bench runs.** The bake-off only needs regex + one LLM candidate to prove/disprove regex sufficiency. Build regex + bench first; add LLM behind the mode switch only if the traps fail; add hybrid only if LLM wins AND token cost matters (YAGNI).

## Data Flow Analysis
- **Source → Sink trace**: gate reads staged files + diff + recipe presence + env → classify → exit code. Correct for pre-commit. FAILS for the marker/clear signal: the "already-QA'd" source (`existingBody`/PR) does not exist at commit time, so the sink (SILENT) is unreachable on a new ship → always-ADVISE (Critical 1).
- **Implicit assumptions**: PR body readable at commit time (false); commit trailer readable at pre-commit (false); repo-level single recipe (monorepo is per-package — EC14).
- **Layout alignment**: recipe/root resolution must read the consumer cwd/`guard.config.json` per `synced-assets-layout-agnostic`; in a monorepo the hook cd's into pkgRel so `.qavis/recipe.json` must resolve there, not the git root.

## Configuration Matrix
| Configuration | Expected | Proposed | Correct? |
|---|---|---|---|
| single-pkg, UI diff, recipe, normal commit | stderr advisory, exit 0 | advisory, exit 0 | Yes |
| single-pkg, UI diff, recipe, `devkit ship` | satisfiable soft-block | exit 3 ALWAYS, clear only via override | **No (Crit 1)** |
| repo with no `.qavis/recipe.json` | SILENT | SILENT | Yes |
| monorepo, UI in pkg A (has recipe) | advise scoped to A | recipe maybe read at git root | Unverified (EC14) |
| consumer without qavis at all | dead-weight SILENT gate | SILENT but always installed | Yes, but see frame |
| LLM outage under strict | must not block ship | exit 2 fail-open | Yes (intended) |

## Missing Considerations
- Pure deletions, binary assets, empty diff, renames / extension changes (EC11-15).
- Override env propagation through the rtk-rewritten ship invocation (must be exported — EC5).
- No `devkit doctor`/drift entry for the new guard; no `sync` propagation story.

## What's Good
- Bench-first sequencing and the import-classifier-from-gate no-drift pattern match sentry/reviewer house style exactly.
- The regex mirror is faithful and the adversarial trap taxonomy is genuinely the right stress set.
- `no-recipe → SILENT` is the correct not-nag default (surface "add a recipe" once in doctor, not per-commit).
- Choosing exit-code over stderr for the agent-visible channel is aligned with the codebase's own rulings.

## Recommended Path Forward
1. **Make the ship block satisfiable, or don't block.** Bring the staged-diff-SHA pass-receipt into v1 (mirror review-cache) so running qavis clears the block; OR ship qavis-advisory WARN-by-default (exit 0 + stderr) like completeness and defer exit-3 until the receipt exists.
2. **Re-home the qavis coupling (frame).** devkit is a portable gate-engine that ships into repos it doesn't own; a qavis-specific classifier is dead weight for every non-qavis consumer and duplicates qavis's regex. The aligned pattern is the **fallow precedent** (husky-block.mts:258 `command -v fallow && fallow audit`): let **qavis expose a cheap `qavis route --diff` dry-run** (ADVISE/SILENT, 0 tokens, pure route.ts — it has none today, only `qavis qa` which launches) and have devkit chain it fail-open. That kills the regex duplication and the coupling in one move. If you keep it devkit-side, justify it explicitly against zero-consumer-tool-deps.
3. **Fix the bench `--fail`** to the flip-table + hard-floors + hashes contract (template: gate-engine/review/eval/bench.mts), not sentry's aggregate delta.
4. Complete the wiring list (AI_GUARD_IDS + STANDALONE_GATES + GUARD_IDS + descriptors + doctor), give the exit-3 fragment its own copy, drop the pre-commit trailer override, force diff prefixes, and fix the `--annotate` remedy string.
